一次M1卡数据分析

2019-07-13 NFC

在一次偶然的机会下，我解开了它的秘密...........

开始工作

数据的获取

我使用ACR-122U，mfoc ，MI6 ，MIFARE，完成了M1卡数据收集 这是收集到的几组数据

这是MI6手机上的读卡软件

这是我苹果手机上用到的开发者计算器

开始分析工作

4位为一组，发现第一组没有变化，而第二组就开始有变化了，根据常用的套路，我确定这是倒序的金额

`

D200 = 2.1￥

F24E = 202.1￥

DE49 =189.1￥

4E48 =185.1￥

`

转化规律是修改数据位置，然后进行16进制转10进制

ABCD → CDAB

开始实例

D200 → 00D2 → 进制转化

进制转化可以使用网页版的进制转化，而我是使用苹果手机的开发者计算器 0是无意义字符，所以打不出来，DEC栏处就是我们需要的十进制数字，与我们的计数方法不一样，他是直接把小数点去掉，扩大百倍 。210缩小百倍后，就是我们现在金额2.1￥

开始实例 NO.2

ABCD → CDAB

F24E → 4EF2 →进制转化

DEC栏里面有我们需要的数据，缩小百倍，就是我们的202.1￥

我把数据分成3处，前面那一处是金额，中间那一处是机器编号，最后一处是校验位。 中间的数据我分析了好久好久。在一次偶然的机会下，我发现这中间这处的数据有时候会变化，有时候不变(重点)，收集了很多很多组数据后，我发现这只是记录最后一次消费机器的编号和消费次数记录。

校验位

当时的这个校验位计算想起来也包含了太多运气，那时候用的是网页版本。 一次偶然，就随便(运气)把数据填到所谓的校验位计算

3B30 D200 00710006900100000000 C55F 2.1￥

看图片，这个校验位就符合CRC-CCIT(XModem)

数据 NO.2

3B30 F24E 00720006900100000000 9114 202.1￥

这组也符合CRC-CCIT(XModem)

点我，进入校验位计算

得出结论

第一组 3B30 不可变

第二组 金额使用倒序，可以自行算出金额以及修改数据

中间一大段是最后一次使用消费机器的编号以及消费次数，

最后一组 校验位，编辑好前面几组数据后根据CRC-CCIT(XModem)算出校验位

动手实践

开始构造一条 98.54￥ 的数据

3B30 照抄

金额部分 98.54￥ → 9854 → 十进制转十六进制 → 【ABCD→CDAB】

如图，转化完成后是 267E ，倒序换后是 7E26

我们有了两组数据 3B30 7E26 ，接下来把消费机器编号的旧数据抄上

3B30 D200 00710006900100000000 C55F 2.1￥

数据已经构造好大半了

3B30 7E26 00710006900100000000

接下来是校验位计算

点我进入校验计算网站

得出4CAC，抄上。注意勾选hex，因为默认是ASCII，我们是数据是十六进制，所以需要勾选成hex。

3B30 7E26 00710006900100000000 4CAC

数据构造完成！

数据分析 NO.2

下面开始分析另一种算法

图中的数据虽然是手动构造的，但他也有分析与学习的价值。 第一处就是金额，方法也和上面的一样，ABCD → CDAB 位置的颠倒与进制的转化，例： 第一组数据的金额 E803， 先 根据 ABCD → CDAB 把E803变成03E8 检验： 我们先打开进制转换的网站勾选好进制选项。

缩小百倍后正好是我们的10￥

第二处的数据我也是在一次偶然的机会找到解决方法

关键是第 2，4组数据 ，金额位里面有重复字符 ，而且第二处的数据里面也出现重复字符，我肯定这不是巧合，是可以合在一起的

D007 对应 2FF8 A00F 对应 5FF0

思考良久后我尝试加一起，

D007 + 2FF8 = FFFF A00F + 5FF0 = FFFF

我把其他组的数据也试着加一起，然后成功了，每一组的第一处和第二处相加必定等于FFFF

第三组的数据和第一组数据一样

动手时间

下面我们尝试构造一条97.14￥的数据

97.14￥ → 9714 → 进制转化 → 颠倒排列 → 运算第二处的数据 →把不变化的数据也抄上

把得到的结果变换一下位置

25F2→F225

运算第二处数据

FFFF - F225 =0DDA

把第三组数据和不变的数据抄上就可以得到完整数据了

F225 ECFF 0DDA 1300 F225 ECFF00FF00FF
